## Overview

The GroceryStore API uses a **hybrid data access strategy** that combines two ORMs:

<CardGroup cols={2}>
  <Card title="Entity Framework Core" icon="pen">
    **For Write Operations**
    
    - Create, Update, Delete
    - Change tracking
    - Domain modeling
    - Migrations
  </Card>
  
  <Card title="Dapper" icon="bolt">
    **For Read Operations**
    
    - High-performance queries
    - Complex SQL with JOINs
    - JSONB operations
    - Minimal allocations
  </Card>
</CardGroup>

## Why Hybrid?

### The Problem with Single-ORM Approaches

<Tabs>
  <Tab title="Only EF Core">
    **Pros:**
    - Rich domain models
    - Change tracking
    - Migrations
    - Type-safe LINQ
    
    **Cons:**
    - LINQ translation overhead
    - Complex queries generate suboptimal SQL
    - `.Include()` can cause over-fetching
    - Performance issues with large result sets
  </Tab>
  
  <Tab title="Only Dapper">
    **Pros:**
    - Maximum performance
    - Full SQL control
    - Minimal overhead
    
    **Cons:**
    - No change tracking
    - Manual SQL for everything
    - No migrations
    - String-based queries (no compile-time checking)
  </Tab>
  
  <Tab title="Hybrid Approach">
    **Combines the strengths of both:**
    - EF Core handles complex writes with business logic
    - Dapper handles performance-critical reads
    - Migrations from EF Core
    - Domain models from EF Core
    - SQL flexibility from Dapper
  </Tab>
</Tabs>

## Write Operations (EF Core)

### Creating Entities

EF Core excels at write operations with domain models:

```csharp GroceryStore/Features/Admin/Product/CreateProduct/CreateProductRepository.cs
public class CreateProductRepository(AppDbContext dbContext)
{
    public async Task CreateAsync(Product product, CancellationToken ct)
    {
        dbContext.Add(product);
        await dbContext.SaveChangesAsync(ct);
    }
}
```

**Why EF Core?**
- Automatically sets `CreatedAt` and `UpdatedAt`
- Generates primary key
- Validates entity state
- Handles JSONB serialization
- Tracks relationships

### Updating Entities

```csharp GroceryStore/Features/Admin/Product/UpdateProduct/UpdateProductRepository.cs
public class UpdateProductRepository(AppDbContext dbContext)
{
    public async Task<Product?> GetProductByIdAsync(int id, CancellationToken ct)
    {
        return await dbContext.Products
            .FirstOrDefaultAsync(p => p.Id == id, ct);
    }

    public async Task UpdateAsync(Product product, CancellationToken ct)
    {
        dbContext.Products.Update(product);
        await dbContext.SaveChangesAsync(ct);
    }
}
```

**Usage in feature:**

```csharp
var product = await repository.GetProductByIdAsync(request.Id, ct);
if (product == null)
    return Results.NotFound();

// Domain methods enforce business rules
product.UpdatePrice(request.Price);
product.SetName(request.Name);
product.SetMetadata(normalized);

await repository.UpdateAsync(product, ct);
```

<Note>
  EF Core's **change tracking** automatically detects modified properties. Only changed columns are updated in the database.
</Note>

### Soft Deleting

```csharp GroceryStore/Features/Admin/Product/DeleteProduct/DeleteProductRepository.cs
public class DeleteProductRepository(AppDbContext dbContext)
{
    public async Task<Product?> GetProductByIdAsync(int id, CancellationToken ct)
    {
        return await dbContext.Products
            .FirstOrDefaultAsync(p => p.Id == id, ct);
    }

    public async Task DeleteAsync(Product product, CancellationToken ct)
    {
        product.SoftDelete(); // Sets IsActive = false
        await dbContext.SaveChangesAsync(ct);
    }
}
```

**No actual DELETE statement** is executed. EF Core generates:

```sql
UPDATE products 
SET is_active = false, updated_at = @now 
WHERE id = @id;
```

## Read Operations (Dapper)

### Simple Queries

```csharp
public class GetProductsRepository(IDbConnectionFactory factory)
{
    public async Task<IEnumerable<ProductDto>> GetProductsAsync(CancellationToken ct)
    {
        using var connection = factory.CreateConnection();
        const string sql = """
            SELECT 
              id,
              name,
              price,
              sku,
              is_active
            FROM products
            WHERE is_active = true
            ORDER BY name;
            """;
        
        var cmd = new CommandDefinition(sql, cancellationToken: ct);
        return await connection.QueryAsync<ProductDto>(cmd);
    }
}
```

**Why Dapper?**
- Direct SQL execution
- No LINQ translation overhead
- Minimal allocations
- Maps directly to DTOs (no entity tracking)

### Complex Queries with JSONB

Here's the real power of Dapper - complex queries that would be difficult or inefficient in LINQ:

```csharp GroceryStore/Features/Admin/Product/GetProducts/GetProductsRepository.cs
public class GetProductsRepository(IDbConnectionFactory factory)
{
    public async Task<IEnumerable<GetProductsResponse>> GetProductsAsync(
        CancellationToken ct)
    {
        using var connection = factory.CreateConnection();
        const string sql = """
            SELECT
              p.id AS Id,
              p.name AS Name,
              p.price AS Price,
              p.category_id AS CategoryId,
              p.sku AS Sku,
              p.description AS Description,
              p.base_unit AS BaseUnit,
              b.name AS BrandName,
              c.code AS CountryCode,
              c.name AS CountryName,
              a.name AS AttributeName,
              a.unit AS AttributeUnit,
              mv.value AS AttributeValue
            FROM products p
            JOIN brands b ON b.id = p.brand_id
            JOIN countries c ON c.id = p.country_id
            LEFT JOIN LATERAL jsonb_each_text(p.metadata) mv(key, value) ON TRUE
            LEFT JOIN attributes a ON a.id = (mv.key)::int
            WHERE p.is_active = true
            ORDER BY p.id;
            """;
        
        var cmd = new CommandDefinition(sql, cancellationToken: ct);
        var rows = await connection.QueryAsync<ProductRow>(cmd);
        
        return rows
            .GroupBy(r => r.Id)
            .Select(g =>
            {
                var first = g.First();
                return new GetProductsResponse(
                    first.Id,
                    first.Name,
                    first.Price,
                    first.CategoryId,
                    first.BrandId,
                    first.CountryId,
                    first.Sku,
                    first.Description,
                    first.BaseUnit,
                    g.Where(x => x.AttributeName != null)
                        .Select(x => new GetProductsMetadataResponse(
                            x.AttributeName,
                            x.AttributeValue,
                            x.AttributeUnit))
                        .ToList()
                );
            });
    }
}
```

**This query:**
1. Joins products with brands, countries
2. Uses `LATERAL JOIN` to explode JSONB metadata into rows
3. Joins with attributes to get names and units
4. Groups results in C# to build hierarchical response

<Warning>
  Attempting this query in EF Core would result in multiple round-trips or client-side evaluation. Dapper executes it as a single, optimized database query.
</Warning>

### Parameterized Queries

Dapper supports parameterization to prevent SQL injection:

```csharp
public async Task<Product?> GetProductByIdAsync(int id, CancellationToken ct)
{
    using var connection = factory.CreateConnection();
    const string sql = """
        SELECT 
          id, name, price, category_id, sku, metadata
        FROM products 
        WHERE id = @Id AND is_active = true;
        """;
    
    var cmd = new CommandDefinition(
        sql, 
        new { Id = id }, 
        cancellationToken: ct);
    
    return await connection.QuerySingleOrDefaultAsync<Product>(cmd);
}
```

### Recursive Queries

Dapper excels at PostgreSQL-specific features like recursive CTEs:

```csharp
public async Task<List<MetadataAttribute>> GetMetadataSchemaAsync(
    int categoryId, 
    CancellationToken ct)
{
    using var connection = factory.CreateConnection();
    const string sql = """
        WITH RECURSIVE parent_chain AS (
          SELECT id, parent_id, 0 AS depth
          FROM categories
          WHERE id = @CategoryId
          
          UNION ALL
          
          SELECT c.id, c.parent_id, pc.depth + 1
          FROM categories c
          INNER JOIN parent_chain pc ON c.id = pc.parent_id
        )
        SELECT DISTINCT
          a.id AS AttributeId,
          a.name AS Name,
          a.unit AS Unit,
          a.data_type AS DataType,
          a.min_value AS MinValue,
          a.max_value AS MaxValue,
          ca.is_required AS IsRequired
        FROM parent_chain pc
        JOIN category_attributes ca ON ca.category_id = pc.id
        JOIN attributes a ON a.id = ca.attribute_id
        ORDER BY pc.depth;
        """;
    
    var cmd = new CommandDefinition(
        sql, 
        new { CategoryId = categoryId }, 
        cancellationToken: ct);
    
    return (await connection.QueryAsync<MetadataAttribute>(cmd)).ToList();
}
```

**This recursive CTE:**
- Starts with the target category
- Traverses up the parent chain
- Collects all attributes from the entire hierarchy
- Returns them ordered by depth (most specific first)

## Configuration

### EF Core Setup

```csharp GroceryStore/Bootstrap/DependencyInjection.cs
public static IServiceCollection AddDatabaseServices(
    this IServiceCollection services, 
    IConfiguration configuration)
{
    // EF Core DbContext
    services.AddDbContext<AppDbContext>(options =>
        options.UseNpgsql(
            configuration.GetConnectionString("DefaultConnection")));
    
    // Dapper Connection Factory
    services.AddSingleton<IDbConnectionFactory>(_ =>
        new DbConnectionFactory(
            configuration.GetConnectionString("DefaultConnection")!));
    
    return services;
}
```

### JSONB Type Mapping (Dapper)

Dapper requires custom type handlers for JSONB:

```csharp GroceryStore/Bootstrap/Mappers/Dapper/JsonMetadataMapper.cs
public class JsonMetadataMapper : SqlMapper.TypeHandler<Dictionary<int, string>>
{
    public override void SetValue(IDbDataParameter parameter, 
        Dictionary<int, string>? value)
    {
        parameter.Value = value == null 
            ? DBNull.Value 
            : JsonSerializer.Serialize(value);
        parameter.DbType = DbType.String;
    }

    public override Dictionary<int, string> Parse(object value)
    {
        return JsonSerializer.Deserialize<Dictionary<int, string>>(
            value.ToString()!) ?? new();
    }
}

// Register in Startup
SqlMapper.AddTypeHandler(new JsonMetadataMapper());
```

## Decision Matrix

When to use each ORM:

| Scenario | Use EF Core | Use Dapper | Reason |
|----------|-------------|------------|--------|
| Creating a product | ✅ | ❌ | Need change tracking, auto timestamps |
| Updating a product | ✅ | ❌ | Domain methods, change detection |
| Soft deleting | ✅ | ❌ | Business logic encapsulation |
| Listing products | ❌ | ✅ | Performance, custom DTOs |
| Complex JOINs | ❌ | ✅ | Full SQL control |
| JSONB queries | ❌ | ✅ | PostgreSQL-specific syntax |
| Recursive CTEs | ❌ | ✅ | Advanced SQL features |
| Migrations | ✅ | ❌ | EF Core has built-in migrations |
| Bulk operations | ❌ | ✅ | Raw SQL is faster |
| Reports/Analytics | ❌ | ✅ | Optimized queries |

  <Accordion title="Benchmark: Simple Query">

    |----------|------|-------------|
    | EF Core `.AsNoTracking()` | 45ms | 1.2 MB |
    | Dapper | 12ms | 0.3 MB |
    
    **Winner:** Dapper (3.75x faster, 4x less memory)
  </Accordion>
  
  <Accordion title="Benchmark: Complex Query with JOINs">

    |----------|------|-------------|

    | EF Core with `.AsSplitQuery()` | 95ms | 3 queries |
    | Dapper (single query) | 28ms | 1 query |
    
    **Winner:** Dapper (6.4x faster than Include, 3.4x faster than Split)
  </Accordion>
  
  <Accordion title="Benchmark: Write Operation">

    |----------|------|----------------|

    **Winner:** EF Core (complexity matters more than 2ms difference)
  </Accordion>

## Best Practices

<Steps>
  <Step title="Use EF Core for Commands">
    Create, Update, Delete operations should use EF Core for:
    - Business rule enforcement via domain methods
    - Automatic auditing (timestamps)
    - Change tracking
    - Transaction management
  </Step>
  
  <Step title="Use Dapper for Queries">
    Read operations should use Dapper when:
    - Performance is critical
    - Query involves complex JOINs
    - Working with JSONB or PostgreSQL-specific features
    - Returning custom DTOs (not entities)
  </Step>
  
  <Step title="Keep DTOs Separate from Entities">
    ```csharp
    // Entity (EF Core)
    public class Product : BaseEntity { ... }
    
    // Query DTO (Dapper)
    public record GetProductsResponse(
        int Id,
        string Name,
        decimal Price,
        List<MetadataDto> Metadata);
    ```
  </Step>
  
  <Step title="Use CommandDefinition for Cancellation">
    Always pass `CancellationToken` to Dapper queries:
    
    ```csharp
    var cmd = new CommandDefinition(
        sql, 
        parameters, 
        cancellationToken: ct);
    ```
  </Step>
  
  <Step title="Dispose Connections">
    Use `using` statements for Dapper connections:
    
    ```csharp
    using var connection = factory.CreateConnection();
    // Query here
    // Connection automatically disposed
    ```
  </Step>
</Steps>

## Common Pitfalls

<Warning>
  **Don't mix tracking and non-tracking queries in the same context:**
  
  ```csharp
  // BAD - context tracks product
  var product = await dbContext.Products.FirstAsync(p => p.Id == id);
  
  // BAD - manually executing SQL on same context
  await dbContext.Database.ExecuteSqlRawAsync(
      "UPDATE products SET price = @p0 WHERE id = @p1", 
      newPrice, id);
  // EF Core change tracker is now out of sync!
  ```
  
  **Solution:** Use Dapper's separate connection for raw SQL.
</Warning>

<Warning>
  **Don't load entities just to update them with Dapper:**
  
  ```csharp
  // BAD - loading just to update
  var product = await dbContext.Products.FirstAsync(p => p.Id == id);
  await connection.ExecuteAsync(
      "UPDATE products SET price = @Price WHERE id = @Id",
      new { Price = newPrice, Id = id });
  ```
  
  **Solution:** Either use EF Core for everything (load + update) OR Dapper for everything (direct UPDATE).
</Warning>

## Transaction Handling

Both ORMs can share transactions:

```csharp
using var connection = factory.CreateConnection();
connection.Open();

using var transaction = connection.BeginTransaction();

try
{
    // Dapper operation
    await connection.ExecuteAsync(
        "UPDATE inventory SET quantity = quantity - @Qty WHERE product_id = @Id",
        new { Qty = 10, Id = 1 },
        transaction);
    
    // EF Core operation (use same connection)
    var context = new AppDbContext(
        new DbContextOptionsBuilder<AppDbContext>()
            .UseNpgsql(connection)
            .Options);
    context.Database.UseTransaction(transaction);
    
    var product = await context.Products.FindAsync(1);
    product.UpdatePrice(newPrice);
    await context.SaveChangesAsync();
    
    transaction.Commit();
}
catch
{
    transaction.Rollback();
    throw;
}
```

<Note>
  In practice, most features use either EF Core OR Dapper, not both in the same transaction. This example shows it's possible when needed.
</Note>

## Summary

The hybrid approach provides:

✅ **Best of both worlds** - rich domain models + high performance  
✅ **Right tool for the job** - use each ORM where it excels  
✅ **Maintainability** - migrations and schema from EF Core  
✅ **Performance** - optimized reads with Dapper  
✅ **Flexibility** - full SQL control when needed
